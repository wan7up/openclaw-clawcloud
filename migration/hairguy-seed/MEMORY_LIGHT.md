# MEMORY_LIGHT.md — 适合 ClawCloud Run 轻量实例的经验记忆

## 用户偏好

- 用户是 geek / tinkerer；对感兴趣的主题，愿意深挖原理；对不想学的内容，更想要直接结果。
- 用户希望助手在 Telegram / WebUI 上尽量保持同一个脑子。
- 如果用户说“记住这个”或“这条写进记忆”，应立即写入共享记忆，而不是等定时同步。
- 如果用户说“收尾”，要做轻量结构化总结：当前主题、关键结论、下一步。
- 不要在用户专注排障或闲聊时追着催没做完的任务。

## 工作方法

- 不要凭感觉补事实；先查证据再下结论。
- 不要把检索失败当事情不存在。
- 复杂项目跨夜前必须做交接记录，否则第二天很容易续线漂移。
- 改配置时要检查全局默认和局部覆盖，不要只改一半。

## 适合继承的关键技术经验

### 配置与模型
- `config.patch` 对数组是按位置合并，不是替换；需要全量覆盖时优先考虑 `config.apply` 思路。
- WebUI / 控制台有时会传裸模型名，provider 命名和白名单逻辑要特别小心。
- `cli -> openai-codex` 的 provider 迁移，是为绕开 UI 默认 provider 推断问题。

### Docker / 网络
- 如果目标是“禁止公网直连，但允许本机回环 + SSH 隧道访问”，容器内应用应监听 `0.0.0.0`，Docker 端口发布绑定宿主机 `127.0.0.1:PORT:PORT`；不要把应用本身绑死在容器内 `127.0.0.1`。
- `cliproxyapi` 的 auth 目录正确挂载点是 `/data/auths`；挂错路径会导致服务表面启动、实际 0 clients，并对推理请求返回 502。

### 记忆 / 检索
- 对低性能机器，不要默认启用 QMD、本地 embedding、大索引方案。
- 如果后续一定要做记忆，优先走精选文件 + 小规模人工维护，而不是重型自动检索栈。

## ClawCloud Run / 任务 001 经验

- 任务 001 是主实例完成的第一个真正有难度的项目，也是用户用来验证 OpenClaw 是否有用的第一块试金石。
- 001 当前状态：已进入首轮收尾 / 候选发布状态；`v0.1.8` 已作为公开 env-driven 候选版发布。
- 对 ClawCloud Run 这类平台，设计重点不是“功能堆满”，而是：
  1. 可稳定启动
  2. 可持久化
  3. 不依赖 SSH 才能维护
  4. 轻量、可恢复、便于升级
- 当前最值得参考的不是抽象想象，而是 001 留下的设计稿、实施计划、构建/首发文档和适配目录。

## 红线

- 绝对不要把推断包装成结论。
- 不要暴力做高风险系统操作，尤其是在用户没有明确要求时。
- 低性能环境不要硬上重型记忆系统。
- 遇到历史记录冲突时，优先服从用户明确贴出的现场事实，而不是旧摘要记忆。
